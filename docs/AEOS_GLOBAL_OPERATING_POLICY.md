## ANTIGRAVITY CONTEXT INTELLIGENCE & TOKEN OPTIMIZATION

**Policy Status:** Mandatory — applies to every execution, session, CLI invocation, and SDK integration.

**Authority:** AntiGravity must optimize engineering quality before optimizing token count.

For every task, AntiGravity must execute the following steps in order:

1. **Scan the repository.** Build a complete picture of available knowledge before any reasoning begins.
2. **Build a repository knowledge graph.** Map every RFC, SPEC, skill, source file, schema, contract, and dependency into a queryable graph.
3. **Discover required capabilities.** Determine which engineering domains the task demands before activating any department.
4. **Activate only the necessary departments.** Do not load the full organization. Activate only the departments whose capabilities are required for this specific task.
5. **Retrieve only the relevant skills.** From activated departments, select only the skills that directly apply. Rank by relevance score. Discard the rest.
6. **Retrieve only the required RFC sections.** Load RFC sections that govern the subsystems involved. Do not load unrelated RFCs.
7. **Retrieve only the required SPEC sections.** Load SPEC sections that define the contracts, algorithms, and pipelines for the task. Do not load unrelated SPECs.
8. **Retrieve only the required source files.** Identify source files that must be read, modified, or referenced. Do not load the entire codebase.
9. **Compress context without losing engineering intent.** Eliminate irrelevant context, redundant repository content, duplicate instructions, and unnecessary history. Preserve architecture, correctness, and production quality. Never remove critical engineering information.
10. **Route the optimized context to the selected AI model.** Deliver only the compressed, relevant context to the model. Let the model reason on signal, not noise.
11. **Measure token usage, latency, cost, quality, and retrieval accuracy.** Every execution must produce measurable metrics.
12. **Produce a benchmark report comparing retrieved context versus available context.** Show what was loaded, what was discarded, and why. Quantify the efficiency gain.
13. **Learn from every execution to improve future retrieval and context optimization.** Store retrieval decisions, relevance scores, and outcomes. Feed this into the Learning System.

### What AntiGravity Must Never Do

AntiGravity must **never** optimize by removing critical engineering information.

Prohibited optimizations:

- Removing architecture decisions to save tokens
- Skipping security context to reduce context size
- Omitting RFC constraints because they seem verbose
- Truncating SPEC contracts to fit a model limit
- Dropping production readiness requirements for speed
- Ignoring test coverage requirements for brevity

Permitted optimizations:

- Eliminating skills from departments not relevant to the task
- Removing RFC sections whose subsystems are not involved
- Dropping SPEC sections whose contracts do not apply
- Excluding source files that are not referenced or modified
- Compressing duplicate instructions into a single canonical form
- Removing conversation history that does not affect the current task

### Mandatory End-of-Execution Metrics

Every execution must conclude with a metrics block covering:

| Metric | Description |
|--------|-------------|
| **Context Utilization** | Percentage of retrieved context actually used in reasoning |
| **Retrieval Accuracy** | Percentage of retrieved items that were relevant to the task |
| **Token Efficiency** | Tokens consumed vs. tokens available in the full repository |
| **Engineering Completeness** | Percentage of required engineering disciplines covered |
| **Production Readiness** | Pass/fail against the Definition of Done checklist |
| **Benchmark Delta** | Comparison of this execution against the previous baseline |
